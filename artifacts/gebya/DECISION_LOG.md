## 2026-06-11 — Owner/Staff Onboarding & Shop Sync Foundation
- **Decision**: Lock the first Codex slice to **onboarding + shop identity only** (slim, multi-device-ready), not the full event model. Ship a server-backed `shops`, `staff`, `devices`, and a `join_code`/`invite_link` foundation. Defer cross-device event sync to a follow-up slice, but do it on a foundation that already supports it.
- **Reason**: The current code is **solo-owner + same-device actor switcher** (verified: `OnboardingScreen.jsx` has no "join existing shop" branch; `lib/api-spec/openapi.yaml` only contains `/healthz`; `lib/db` schema is empty). The single highest-leverage gap for the field test is: a staff member's phone must be able to *belong to the owner's shop*. Until that exists, "owner sees staff sales" is impossible regardless of how polished the local actor switcher is. The event-sync layer adds little value on top of a join layer that doesn't exist.
- **What was rejected**:
  - "Onboarding + sale sync" → too thin; without a join layer, owner cannot see staff sales at all. Tempting because it ships fast, but it does not unblock the core field-test question.
  - "Onboarding + flexible event model + sale sync" → overbuild. v1 must not invent a `correction_request` event type or a full permissions matrix before a single staff phone has joined a shop. The flexible event model can wait one slice.
  - "Onboarding + sale + Dubie sync" → Dubie is the second-most-valuable field-test event, but it doubles the surface area of the first slice and the join foundation is the bottleneck. Add Dubie in slice 2.
  - Hard permissions matrix now → the user has explicitly warned "Do not add complex permission systems until owner/staff flow is validated." Keep v1 to **role label + capability flags derived from role**, not a free-form policy engine.
  - "Phone is required to join" → rejected. Phone breaks the field test because staff frequently do not have a reliable second phone number; we already have `isValidPhone` and an optional phone field in `OnboardingScreen`.
- **Next check**: After slice 1 ships and at least 2 shop owners have used the join flow on real devices in the field test. Revisit: should we add the flexible event envelope, or stay on `sale` only for another week?

### 2026-06-11 (revised) — Reject identity-only; ship core event sync in slice 1
- **Decision**: Slice 1 is **onboarding + shop identity + staff join + core event sync (sale, customer_payment, customer_credit-owner-only, quick_note) + StaffFeed + Today by Staff + sync chip + bank-trust copy**. Identity-only was correctly challenged as underbuilt for the field-test hypothesis. Slice 1.1 adds expense, device deactivate, correction request, soft void, Telegram owner alerts, notification preferences, Trusted Staff role UI. Slice 2 adds supplier, standalone quick notes, commission, Manager role, advanced perms.
- **Reason**: The field-test question is "can the owner see what staff did?" not "did the phones join?" A join-only slice yields a plumbing milestone, not a product signal. We add 2 new endpoints (`POST /api/events`, `GET /api/events`) and 1 new table (`events`) on top of the identity foundation. Cost is bounded; field-test learning is the difference between a milestone and a product.
- **What was rejected (challenge to my previous answer)**:
  - Identity-only slice → rejected. Produces no product signal and a false-positive "join works, ship it" reading.
  - "Add envelope, sync only sale, defer Dubie" → rejected. The staff feed is more useful with both money-in (sale) and money-in (payment) for the owner to reconcile against the drawer. Sale-only is a worse field-test wedge than sale + payment. Dubie `customer_credit` is included in the envelope but **owner-only in slice 1** (Basic Staff cannot create it). Trusted Staff + Dubie is slice 1.1.
  - "One event type per code path" → rejected. We ship **one envelope** with `type` as a discriminator and zod-validated `payload` per type. This keeps the door open for slice 1.1 types without rewriting.
  - "Server trusts client role" → rejected. Server re-checks role on every `POST /api/events` against the role-permission matrix. Client is best-effort; server is the gate.
- **Dubie role carve-out (Option A)**: Basic Staff can READ owner-created Dubie in the staff feed but cannot create one. The E permission matrix governs slice 1. Trusted Staff + Dubie is a clean *new permission* in slice 1.1, not a tightening of an already-shipped one. (Owner can override this by promoting the staff to Trusted Staff earlier from Settings if the field test demands it.)
- **`actor_role_at_event` and `actor_name_snapshot` are non-negotiable.** They lock identity at the moment of the event so role demotions or staff deactivation never rewrite the audit trail. Field-test trust invariant.
- **Next check**: After 2 shop owners have run the field test for 1 week with real staff phones, evaluate: (a) is the staff feed the wedge that drives retention? (b) do we need expense sync before Trusted Staff + Dubie, or after? (c) is the join code friction a real drop-off, and is QR scan the next bet?

### 2026-06-11 (revised again) — Permission model, lifecycle states, and rejoin
- **Decision (Dubie)**: Reject Option A (owner-only) and Option C (all Basic Staff). Ship **Option B: permission-based, default owner-only for Basic Staff, owner-overridable per-staff**. The default for `can_create_customer_credit` is false for Basic Staff; the owner can flip it true for a specific trusted staff from Settings. Trusted Staff / Manager / Owner have it true by default. This matches actual Ethiopian shop reality where the floor salesperson is the one who records Dubie, but the owner gates trust per-staff, not per-role.
- **Decision (lifecycle)**: Two `staff_status` values in slice 1: `active` and `inactive`. **Do not ship `suspended`** in slice 1 — re-activation from `inactive` is one tap and there is no side effect that benefits from a soft-pause. Add `suspended` in slice 1.1 only if the field test asks for it. Two `device_status` values: `active` and `revoked`. `lost` is slice 1.1.
- **Decision (phone replacement)**: Phone replacement, planned phone change, and same-phone reinstall all go through the **same rejoin flow**. New `device_id` is bound to the existing `staff_id` (matched by `display_name`, case-insensitive, trimmed). Previous device(s) are `revoked`. The owner is notified "Tigist's device changed on <date>." We do not auto-restore identity from a stored token alone — without a server-side check (password/OTP/biometric), that is not a trustworthy identity signal in a theft scenario.
- **Decision (rejoin binding)**: Rejoin is bound by **display name match**, not phone number. Phone is unreliable in Ethiopia (SIM churn, shared phones, no second phone). The join screen reminds the staff: "Use the same name the owner has in their staff list. If you don't remember, ask the owner." This is the only safe design without a password.
- **Decision (deactivation)**: Owner-initiated deactivation is the only departure path in slice 1. Staff-initiated leave is slice 1.1. Deactivation revokes all the staff's devices in the same transaction; the staff phone shows "Your access was ended by the owner" on next open. Local DB is preserved (not silently wiped). Historical events stay attributed to the staff via `actor_staff_member_id` + `actor_name_snapshot` + `actor_role_at_event` — all immutable.
- **Decision (events buffered on a deactivated staff's phone)**: They are lost. The owner sees the staff's "last seen" timestamp in the staff list and acts on it. This is the trade-off of offline-first without accounts; we do not pretend otherwise.
- **Decision (the rejoin conflict case)**: If a staff with the same display name is already active on another device, rejoin returns 409 "This name is already used in this shop on another phone." The owner must deactivate the old device first. This is the defense against a stolen phone being used to rejoin.
- **Pushback I owe the user**: My previous "Option A" answer for Dubie was wrong for the Ethiopian shop context. I justified it with risk language, but the actual shop workflow is owner-off-floor + salesperson-on-floor. Removing the capability creates a paper-plus-notebook workflow, which violates the thesis. The corrected model is "capability default + per-staff override." The role matrix is no longer a 3-column boolean table; it is role defaults + per-staff delta.
- **Out of scope (still hard)**: `suspended` status, `lost` device status, "leave shop" staff flow, push notifications, SMS, email, password/PIN/OTP/biometric, bank integration, money movement, payroll, HR, inventory, accounting, POS, marketplace, credit scoring. None of these.
- **Next check**: After 2 shop owners have run the field test for 1 week with at least 1 staff phone each, evaluate: (a) does the per-staff Dubie override actually get used, or do owners just promote to Trusted Staff wholesale? (b) how often does the rejoin flow actually fire (lost phones vs. new staff)? (c) do we need `suspended` for soft pauses, or is one-tap re-activation enough? (d) is the display-name-match rejoin binding confusing in practice, and is a per-staff rejoin link the right slice 2 bet?
